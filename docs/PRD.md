# Product Requirements Document: ClipSort

**Version:** 1.0
**Date:** 2026-03-10
**Status:** Draft

---

## 1. Overview

ClipSort is a command-line tool, packaged as a Docker container, that helps student filmmakers organize raw video footage into a structured folder hierarchy based on scene and take information. It supports three levels of sophistication:

1. **Filename-based organization** - Parse filenames like `1a`, `1b`, `Scene2_Take1` and sort into scene folders
2. **QR code detection** - Read pre-printed QR codes from clapper boards visible in video frames to auto-identify scenes
3. **Visual clapper board OCR** - Detect and read text directly from physical clapper boards in video

## 2. User Personas

### Primary: Student Filmmaker (Jeremy's mentee)
- Has shot raw footage for a 3-10 minute short film
- Files are named with simple conventions (e.g., `1a.mp4`, `1b.mp4`, `1c.mp4`)
- Wants clips organized into scene folders quickly
- May not have technical expertise beyond running a single command
- Works on macOS or Windows with Docker Desktop installed

### Secondary: Student Film Mentor (Jeremy)
- Helps the student adopt better production workflows
- Wants to introduce QR-coded clapper boards for future shoots
- Values reproducibility and ease of deployment (hence Docker)

## 3. Assumptions and Constraints

- Input video files are common formats: MP4, MOV, MKV, AVI
- The tool operates on a local directory of video files
- Docker is available on the user's machine
- No GPU is required for Phase 1-2; GPU is optional for Phase 3
- Files are not renamed destructively; originals are preserved or moved (user's choice)
- The tool is non-interactive (all options via CLI flags) for scriptability

## 4. Use Cases

### UC-1000: Filename-Based Organization

#### UC-1001: Organize clips by scene number from filename

**Actor:** Student Filmmaker
**Precondition:** A directory contains video files named with a leading scene number (e.g., `1a.mp4`, `1b.mp4`, `2a.mp4`, `2b.mp4`)
**Trigger:** User runs `clipsort organize <input-dir> <output-dir>`
**Flow:**
1. Tool scans the input directory for video files
2. Tool parses each filename to extract a scene identifier
3. Tool creates subdirectories in the output directory named by scene (e.g., `scene_01/`, `scene_02/`)
4. Tool copies (or moves, if `--move` flag is set) each file into its scene folder
5. Tool prints a summary of what was organized

**Postcondition:** Output directory contains scene folders with the corresponding clips
**Acceptance Criteria:**
- Files `1a.mp4`, `1b.mp4` end up in `scene_01/`
- Files `2a.mp4`, `2b.mp4`, `2c.mp4` end up in `scene_02/`
- A summary report is printed to stdout
- Original files are untouched unless `--move` is used

#### UC-1002: Support multiple filename conventions

**Actor:** Student Filmmaker
**Precondition:** Files may use various naming patterns
**Trigger:** User runs `clipsort organize <input-dir> <output-dir>`
**Flow:**
1. Tool attempts to match filenames against known patterns in priority order:
   - `{scene}{take_letter}` (e.g., `1a`, `2b`, `12c`)
   - `{scene}_{take}` (e.g., `1_1`, `2_3`)
   - `Scene{scene}_Take{take}` (e.g., `Scene1_Take2`)
   - `S{scene}T{take}` (e.g., `S01T03`)
2. Tool reports which pattern was detected
3. Files that don't match any pattern go to an `unsorted/` folder

**Postcondition:** All files are organized or placed in `unsorted/`
**Acceptance Criteria:**
- Each supported pattern correctly extracts scene number
- Unrecognized files are not silently dropped
- The detected pattern is logged

#### UC-1003: Preview organization without moving files (dry run)

**Actor:** Student Filmmaker
**Precondition:** User wants to verify the organization plan before executing
**Trigger:** User runs `clipsort organize --dry-run <input-dir> <output-dir>`
**Flow:**
1. Tool scans and parses all filenames
2. Tool prints the planned organization (source -> destination) without performing any file operations

**Postcondition:** No files are moved or copied; plan is displayed
**Acceptance Criteria:**
- Output shows each file and its target directory
- No filesystem changes occur
- Exit code is 0

#### UC-1004: Generate an organization report

**Actor:** Student Filmmaker
**Precondition:** Organization has been performed (or dry-run)
**Trigger:** User includes `--report` flag or report is generated by default
**Flow:**
1. Tool generates a summary listing:
   - Number of scenes detected
   - Number of clips per scene
   - Any files that could not be categorized
   - Total files processed

**Postcondition:** Report is printed to stdout and optionally saved to a file
**Acceptance Criteria:**
- Report includes scene count, clips per scene, and unsorted count
- Report can be saved with `--report-file <path>`

#### UC-1005: Handle filename conflicts

**Actor:** Student Filmmaker
**Precondition:** Two files would resolve to the same destination path
**Trigger:** Conflict detected during organize
**Flow:**
1. Tool detects the conflict
2. Tool appends a numeric suffix to avoid overwriting (e.g., `1a_2.mp4`)
3. Tool logs a warning about the conflict

**Postcondition:** No files are overwritten; conflicts are resolved safely
**Acceptance Criteria:**
- No data loss from naming collisions
- Warning is logged for each conflict

#### UC-1006: Recursive directory scanning

**Actor:** Student Filmmaker
**Precondition:** Video files are spread across subdirectories (e.g., from multiple memory cards)
**Trigger:** User runs `clipsort organize --recursive <input-dir> <output-dir>`
**Flow:**
1. Tool recursively scans all subdirectories for video files
2. Organization proceeds as normal

**Postcondition:** All video files from nested directories are organized
**Acceptance Criteria:**
- Files from arbitrarily nested directories are found
- Only recognized video file extensions are processed

### UC-2000: QR Code-Based Scene Detection

#### UC-2001: Detect QR code from clapper board in video

**Actor:** Student Filmmaker
**Precondition:** Videos begin with a clapper board displaying a QR code that encodes scene/take info
**Trigger:** User runs `clipsort detect <input-dir> <output-dir>`
**Flow:**
1. Tool opens each video file
2. Tool samples frames from the first N seconds (configurable, default 10s)
3. Tool scans sampled frames for QR codes
4. If a QR code is found, tool decodes the scene/take information
5. Tool organizes the clip based on decoded information

**Postcondition:** Clips are organized by the scene info from QR codes
**Acceptance Criteria:**
- QR codes are detected within the first 10 seconds of video
- Decoded scene information matches expected format
- Fallback to filename-based if no QR code is found

#### UC-2002: Define QR code format

**Actor:** Mentor / Production Coordinator
**Precondition:** User needs to print QR codes for clapper boards
**Trigger:** User runs `clipsort qr-generate --scene 1 --take 1`
**Flow:**
1. Tool generates a QR code image encoding structured data:
   ```json
   {"scene": 1, "take": 1, "project": "short-film"}
   ```
2. QR code is saved as a PNG file suitable for printing

**Postcondition:** Printable QR code image is created
**Acceptance Criteria:**
- QR code is large enough to be readable on camera (minimum 3cm x 3cm at 300 DPI)
- QR code decodes back to the correct scene/take data
- Optional project name field is supported

#### UC-2003: Batch-generate QR codes for all scenes

**Actor:** Mentor / Production Coordinator
**Precondition:** Shot list is known (e.g., 5 scenes, up to 10 takes each)
**Trigger:** User runs `clipsort qr-generate --scenes 5 --takes 10`
**Flow:**
1. Tool generates QR code images for all scene/take combinations
2. Tool creates a printable PDF sheet with all QR codes labeled

**Postcondition:** PDF with all QR codes is ready to print
**Acceptance Criteria:**
- PDF contains all scene/take combinations
- Each QR code is labeled with its scene and take number
- PDF is formatted for standard letter or A4 paper

#### UC-2004: Handle videos without QR codes gracefully

**Actor:** Student Filmmaker
**Precondition:** Some videos have QR-coded clappers, others don't
**Trigger:** User runs `clipsort detect <input-dir> <output-dir>`
**Flow:**
1. Tool attempts QR detection on all videos
2. For videos without detected QR codes, tool falls back to filename-based organization
3. Videos that can't be categorized by either method go to `unsorted/`

**Postcondition:** All videos are handled; none are silently dropped
**Acceptance Criteria:**
- Mixed input (some with QR, some without) is handled correctly
- Fallback chain: QR -> filename -> unsorted
- Report indicates which method was used for each file

### UC-3000: Visual Clapper Board Detection and OCR

#### UC-3001: Detect clapper board presence in video frames

**Actor:** Student Filmmaker
**Precondition:** Videos begin with a visible clapper board (standard black-and-white slate)
**Trigger:** User runs `clipsort detect --mode ocr <input-dir> <output-dir>`
**Flow:**
1. Tool samples frames from the beginning of each video
2. Tool uses image analysis to detect the presence of a clapper board (high-contrast striped pattern, rectangular shape)
3. If detected, tool extracts the region of interest containing the slate text

**Postcondition:** Clapper board regions are identified for OCR
**Acceptance Criteria:**
- Clapper boards are detected with reasonable accuracy (>80% on well-lit slates)
- Detection works on standard black-and-white clapper boards
- Processing time is under 30 seconds per video

#### UC-3002: Read scene/take info from clapper board via OCR

**Actor:** Student Filmmaker
**Precondition:** Clapper board has been detected in a frame
**Trigger:** Continuation of UC-3001
**Flow:**
1. Tool preprocesses the detected slate region (contrast enhancement, deskewing)
2. Tool runs OCR on the preprocessed image
3. Tool extracts scene and take numbers from OCR results
4. Tool organizes the clip based on extracted info

**Postcondition:** Clips organized by OCR-extracted scene info
**Acceptance Criteria:**
- Scene number is correctly extracted from legible handwriting or printed text
- Confidence score is reported for each OCR result
- Low-confidence results trigger a warning and fallback

#### UC-3003: Split a continuous video file at clapper board boundaries

**Actor:** Student Filmmaker
**Precondition:** A single long video file contains multiple scenes separated by clapper boards
**Trigger:** User runs `clipsort split <input-file> <output-dir>`
**Flow:**
1. Tool scans through the entire video, sampling frames periodically
2. Tool detects clapper board appearances (visual or QR)
3. Tool identifies timestamps where clapper boards appear
4. Tool splits the video at those timestamps using FFmpeg (stream copy for speed)
5. Each resulting clip is named based on the detected scene/take info

**Postcondition:** Single video is split into individual scene clips
**Acceptance Criteria:**
- Split points are within 1 second of actual clapper appearance
- Splits use stream copy (no re-encoding) when possible
- Each output clip has a meaningful filename
- Original file is not modified

#### UC-3004: Audio-based clap detection as supplementary signal

**Actor:** Student Filmmaker
**Precondition:** Clapper board produces an audible "clap" sound
**Trigger:** Used internally as part of the detection pipeline
**Flow:**
1. Tool extracts audio track from video
2. Tool analyzes audio for sharp transient peaks (clap sounds)
3. Detected audio claps are correlated with visual clapper detections for higher accuracy

**Postcondition:** Clap timestamps are available as supplementary detection data
**Acceptance Criteria:**
- Audio claps are detected with >70% accuracy
- Audio detection supplements (not replaces) visual detection
- False positive rate is manageable (< 20%)

### UC-4000: Docker and Deployment

#### UC-4001: Run tool via Docker with mounted volumes

**Actor:** Student Filmmaker
**Precondition:** Docker is installed; video files are on the host machine
**Trigger:** User runs:
```
docker run -v /path/to/videos:/input -v /path/to/output:/output clipsort organize /input /output
```
**Flow:**
1. Docker container starts with mounted input and output volumes
2. Tool executes the requested command
3. Results are written to the output volume
4. Container exits

**Postcondition:** Organized files appear on the host filesystem
**Acceptance Criteria:**
- Container runs without requiring special privileges
- Input and output paths are clearly documented
- Container exits with appropriate exit codes (0 = success, 1 = error)

#### UC-4002: Provide helpful error messages

**Actor:** Student Filmmaker
**Precondition:** User provides incorrect arguments or paths
**Trigger:** Various error conditions
**Flow:**
1. Tool validates inputs before processing
2. Clear error messages are displayed for common issues:
   - Missing input directory
   - Empty input directory
   - No video files found
   - Output directory not writable
   - Invalid flags

**Postcondition:** User understands what went wrong and how to fix it
**Acceptance Criteria:**
- Error messages suggest corrective action
- Exit code is non-zero on error
- No stack traces shown to user (only in `--verbose` mode)

#### UC-4003: Display progress during processing

**Actor:** Student Filmmaker
**Precondition:** Processing a batch of video files
**Trigger:** Any command that processes multiple files
**Flow:**
1. Tool displays a progress indicator (file X of Y)
2. For long-running operations (QR scan, OCR), per-file progress is shown

**Postcondition:** User can see processing progress
**Acceptance Criteria:**
- Progress is shown for batches of more than 1 file
- No excessive output; progress updates in place when possible

### UC-5000: Configuration and Extensibility

#### UC-5001: Custom filename pattern via regex

**Actor:** Advanced User
**Precondition:** User's naming convention doesn't match built-in patterns
**Trigger:** User runs `clipsort organize --pattern '{scene}_{take}' <input-dir> <output-dir>`
**Flow:**
1. User provides a pattern string with `{scene}` and `{take}` placeholders
2. Tool converts the pattern to a regex and uses it for parsing
3. Organization proceeds using the custom pattern

**Postcondition:** Files are organized using the custom pattern
**Acceptance Criteria:**
- Custom patterns override default pattern detection
- Invalid patterns produce a clear error message
- `{scene}` placeholder is required; `{take}` is optional

#### UC-5002: Configure output folder naming

**Actor:** Advanced User
**Precondition:** User wants custom folder names (not `scene_01`)
**Trigger:** User runs `clipsort organize --folder-format 'Scene {scene}' <input-dir> <output-dir>`
**Flow:**
1. Tool uses the provided folder format string
2. `{scene}` is replaced with the detected scene number

**Postcondition:** Folders are named according to user's format
**Acceptance Criteria:**
- Default format is `scene_{scene:02d}` (zero-padded)
- Custom format supports `{scene}` placeholder

## 5. Non-Functional Requirements

| ID | Requirement | Target |
|----|-------------|--------|
| NFR-001 | Filename organization of 100 files completes in < 5 seconds | Performance |
| NFR-002 | QR detection processes at least 2 videos per minute | Performance |
| NFR-003 | Docker image size < 2 GB | Deployment |
| NFR-004 | Tool works without internet access once image is built | Offline |
| NFR-005 | No data loss: original files are never modified or deleted without `--move` | Safety |
| NFR-006 | Cross-platform: runs on macOS, Linux, and Windows via Docker | Compatibility |
| NFR-007 | Clear CLI help text via `--help` on every command | Usability |

## 6. Out of Scope (v1)

- GUI / web interface
- Cloud storage integration (Google Drive, S3)
- Video transcoding or format conversion
- Video editing or trimming beyond scene splitting
- Real-time processing / watching a directory for new files
- Multi-camera synchronization
